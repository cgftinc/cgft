# how training works

`train()` handles everything from dataset upload to job launch in a single call. here's a minimal end-to-end example before diving into the internals.

## example: training a string reversal model

a good way to understand the pipeline is a toy task with no external dependencies — reversing a string. no corpus, no search tool, just a model, a dataset, and a reward function.

see [notebooks/reverse_env.ipynb](https://github.com/cgft-io/synthetic-data-prep-lib/blob/main/notebooks/reverse_env.ipynb) for the `ReverseEnv` implementation.

```python
import random, string
from synthetic_data_prep.trainer import train
from my_envs import ReverseEnv

# your dataset: a list of dicts with whatever keys your env expects
dataset = [
    {"question": f"Reverse the following string: {s}", "answer": s[::-1]}
    for s in ("".join(random.choices(string.ascii_letters, k=random.randint(5, 20))) for _ in range(50))
]

experiment_id = train(
    env_class=ReverseEnv,   # your BaseEnv subclass — defines tools, reward, and dataset loading
    env_args={},            # constructor args passed to ReverseEnv(...); omit dataset_path, it's injected automatically
    dataset=dataset,        # dataset is expected to be a list of dicts; it will be serialized to JSONL and uploaded to blob storage
    prefix="reverse",       # namespace prefix for upload paths (e.g., datasets/reverse/<hash>/...)
    api_key=API_KEY,        # from https://app.cgft.io/account/api-keys
)
```

see [notebooks/reverse_env.ipynb](https://github.com/cgft-io/synthetic-data-prep-lib/blob/main/notebooks/reverse_env.ipynb) for the full runnable version.

## what train() does

`train()` does three things in sequence:

**1. upload dataset**

serializes your dataset to JSONL (one JSON object per line) and uploads it to blob storage under `datasets/{prefix}/{content_hash}/`. the path is auto-generated from a hash of the content — so the same dataset always gets the same path and won't be re-uploaded unnecessarily.

**2. bundle and upload environment**

pickles your environment class and packages it with metadata (python version, pip dependencies, constructor args). before uploading, it spins up an isolated subprocess to validate the bundle — this catches import errors or missing dependencies before the training job starts.

the env bundle is versioned by a hash of its contents, so re-uploading the same env is a no-op.

**3. launch experiment**

calls the cgft platform with the uploaded env paths. the training cluster reconstructs your environment from the bundle and starts RL training against your dataset.
