# synthetic data generation

generating question-answer pairs automatically from your corpus for training search models. turns any collection of documents into a training dataset.

## quickstart

see [notebooks/simple_rag_pipeline.ipynb](https://github.com/cgftinc/synthetic-data-prep-lib/blob/main/notebooks/simple_rag_pipeline.ipynb) for a full working example.

```python
from synthetic_data_prep.corpus import CorporaChunkSource
from synthetic_data_prep.qa_generation import generate_dataset
from synthetic_data_prep.envs import CorporaSearchEnv
from synthetic_data_prep.trainer.pipeline import train
import synthetic_data_prep

# 1. Point to dataset
source = CorporaChunkSource(api_key=API_KEY, corpus_name="my-docs", base_url=BASE_URL)
source.populate_from_folder("./samples/posthog")

# 2. Create synthetic QA
dataset = generate_dataset(
    source=source,
    api_key=API_KEY,
    corpus_description="Posthog documentation",
    example_queries=["how to feature flag", "setup reverse proxy"],
    num_single_hop=40,
    num_multi_hop=5,
)

# 3. Launch training
experiment_id = train(
    env_class=CorporaSearchEnv,
    env_args={
        "api_key": API_KEY,
        "corpus_id": source.corpus_id,
        "base_url": BASE_URL,
    },
    dataset=dataset.to_list(),
    api_key=API_KEY,
    local_modules=[synthetic_data_prep],
)
```

for detailed explanations and customization, continue reading below or see the [full notebook](https://github.com/cgftinc/synthetic-data-prep-lib/blob/main/notebooks/synthetic_rag_dataset_gen.ipynb).

## how it works

the process samples chunks from your corpus and uses an LLM to generate realistic search queries that could only be answered by that chunk. this creates `<question, answer, reference_chunks>` triplets for training.

**two question types**:

1. **single-hop** — answer found in one chunk
   - "postgres jsonb index syntax" → finds SQL reference doc
   - simpler, faster to generate (1 LLM call per chunk)

2. **multi-hop** — requires information from multiple chunks
   - "k8s pod limits in production setup" → needs k8s docs + production guide
   - more realistic search behavior (2 LLM calls: find related chunks, then validate connection)

## generation process

### corpus context
first, build corpus-level context to help the LLM generate realistic queries:

```python
from synthetic_data_prep.qa_generation.helpers import render_template
from synthetic_data_prep.qa_generation.response_parsers import parse_corpus_summary_response

# Provide your own context
YOUR_DESCRIPTION = "Posthog documentation including docs, company policy, etc."
EXAMPLE_QUERIES = [
    "how to feature flag",
    "setup reverse proxy to avoid ad blockers",
    "posthog install nextjs",
]

# generate_dataset() handles this automatically, but you can inspect it manually:
# LLM analyzes sampled chunks + your context to generate corpus summary + more example queries
corpus_summary, example_queries = parse_corpus_summary_response(llm_response)
```

providing your own description and example queries helps the LLM understand your corpus theme, domain, user personas, and typical search patterns. the LLM then expands on these to generate more diverse examples.

### single-hop generation

```python
from synthetic_data_prep.qa_generation.helpers import generate_single_hop_batch
from synthetic_data_prep.qa_generation.response_parsers import parse_single_hop_response

dataset = generate_single_hop_batch(
    source=source,
    client=openai_client,
    model="grok-4-fast-non-reasoning",
    system_prompt=single_hop_system_prompt,  # includes corpus context
    user_template=single_hop_user_template,  # chunk + context template
    num_samples=40,
    response_parser=parse_single_hop_response,
    max_questions=2,  # limit questions per chunk for diversity
)

print(dataset.summary())
# QADataset: 67 total data points
#   Single-hop: 67
#   Multi-hop: 0
```

the LLM sees the chunk with before/after context and generates keyword-focused queries like real search behavior.

### multi-hop generation

multi-hop requires finding chunks with meaningful relationships:

```python
from synthetic_data_prep.qa_generation.helpers import generate_multi_hop_batch

dataset = generate_multi_hop_batch(
    source=source,
    client=openai_client,
    model="grok-4-fast-non-reasoning",
    related_query_system_prompt=related_query_prompt,
    related_query_user_template=related_query_template,
    multi_hop_system_prompt=multi_hop_prompt,
    multi_hop_user_template=multi_hop_template,
    num_samples=15,
    related_query_parser=parse_related_queries_response,
    multi_hop_parser=parse_multi_hop_validation_response,
    top_k_bm25=5,  # candidates per query
    top_related_chunks=3,  # chunks to validate per source
)
```

**step 1**: for each chunk, LLM generates BM25 queries to find related chunks (targeting entity names, references, abstractions)

**step 2**: search corpus with those queries, filter adjacent chunks

**step 3**: validate chunk pairs have real dependency (explicit reference or abstraction levels), generate multi-hop questions

multi-hop questions test real search behavior — the initial query finds the first chunk, but that chunk references or relates to information in a second chunk. the model must recognize it needs more context and search again.

dataset structure:
```json
{
  "question": "k8s pod memory limits",
  "answer": "Set resource limits in pod spec...",
  "reference_chunks": [
    {
      "id": "abc123...",
      "metadata": {"file": "k8s-docs.md", "headers": ["Resources"]},
      "content": "To configure memory limits..."
    }
  ],
  "qa_type": "single_hop"
}
```

## quality tips

- **chunk size matters**: 1024-2048 chars works well (too small = low context, too big = noisy)
- **corpus diversity**: sample different doc types for broader question coverage
- **limit questions per chunk**: use `max_questions=2` to avoid over-concentrating on few chunks
- **review samples**: inspect a few generated Q&As before batch processing
- **multi-hop is expensive**: fewer samples (10-20) can still create diverse questions
